### CQRS Pipeline (`cqrs_pattern.md`)
- Pipeline order: `LoggingMiddleware → RetryMiddleware → TransactionMiddleware → Handler`
- Retry wraps Transaction → each retry gets a fresh DB transaction
- Commands opt-in via `options: { transactional: true, retryable: true }`
- TransactionMiddleware uses `ITransactionManager` (abstract) + `AsyncLocalStorage`
- Repositories use `getTx() ?? this.prisma` — zero signature changes

### Database (`database_standard.md`)
- Primary keys: UUID (`@default(uuid())`), NEVER `autoincrement()`
- Naming: camelCase in code, `@map("snake_case")` in DB
- Soft delete: `deletedAt DateTime?` for important models
- Prisma v7: NO `url` in `schema.prisma`, use `prisma.config.ts`
- Port: `15432` (not default 5432)

### Testing (`testing_standard.md`)
- Co-location: `*.spec.ts` next to source file
- Mock pattern: `jest.Mocked<Interface>` with `as unknown as` cast
- ESM libraries: `jest.mock('uuid', () => ({ v7: jest.fn(() => 'mock-uuid') }))`
- Path alias: `@/` mapped via `moduleNameMapper`

### Validation (`zod_validation.md`)
- Zod is the ONLY validation library (no class-validator, no typebox)
- Schema location: `modules/<module>/presentation/schemas/<action>.schema.ts`

### Logging (`logging_standard.md`)
- Dual-layer: HTTP hooks + CQRS LoggingMiddleware
- Use `createLogger(serviceName)` from shared-kernel
- NEVER `console.log`
- **HTTP response shape**: ALWAYS use shared-kernel utilities — `buildErrorBody()`, `buildSuccessBody()`, `httpStatusToCode()`. Never inline `{ success, message, error, meta }` in hooks/filters/interceptors.

### Multi-Tenancy (`multi_tenancy.md`)
- `orgId` BẮT BUỘC trong JWT payload — auth guard reject nếu missing
- Repositories dùng `getTenantId()` từ AsyncLocalStorage — KHÔNG truyền orgId qua parameter
- Mọi DB query phải có `where: { orgId: getTenantId(), ... }` — không có exception trừ system admin
- DB schema: orgId column + `@@index([orgId, ...])` trên mọi domain model
- Org-switch = issue new token với targetOrgId mới

### Event Sourcing (`event_sourcing.md`)
- Chỉ dùng cho Credit và Reputation — không dùng cho CRUD modules
- OCC: `@@unique([aggregateId, version])` → catch P2002 → `OptimisticLockError`
- Projection (read model) update synchronously trong cùng transaction với event append
- Event payload immutable — không modify stored events

### RAG / AI (`rag_ai_integration.md`)
- Circuit Breaker bắt buộc — không call Claude API trực tiếp
- Hybrid search: pgvector (semantic) + Elasticsearch (keyword) → RRF merge
- Per-tenant Elasticsearch index: `knowledge-{orgId}`
- Chunk trước khi embed — không embed toàn bộ document

### Security
- CORS origins from env vars, NEVER `['*']`
- Mandatory: `@fastify/helmet`, `@fastify/compress`, `@fastify/rate-limit`

- **Jest SyntaxError: Unexpected token 'export' khi import ESM từ shared-kernel** `[Unit tests auth-service]`
  → Bỏ isolatedModules trong ts-jest config package.json, dùng diagnostics.ignoreCodes: [151002] thay thế
- **Domain Entity: ensureCanLogin throw error chung chung Error()** `[Domain logic auth-service]`
  → Tạo UserCannotLoginError trong auth.error.ts và sửa lại entity
- **Entity data loss: rehydrate() method in User entity drops profile data** `[auth-service Domain Entity rehydration]`
  → Update the rehydrate() factory method to include profile in the recreated instance
- **Architecture: Microservice registered with wrong name 'api-gateway' instead of 'core-api'** `[core-api microservice communication]`
  → Fix service name registration in app.module.ts ClientsModule to match actual service name
- **Basic CommandBus lacks cross-cutting concerns (Middleware) like Logging/Transactions, leading to duplicated code in handlers.**
  → Implemented Onion Architecture Middleware Pipeline in CommandBus. execute(command) now uses recursive dispatch. Created ICommandMiddleware interface. Added LoggingMiddleware as proof of concept using the central logger.
- **Prisma transactions inside CQRS Middlewares break Hexagonal Architecture because Repositories require the 'tx' transaction client parameter explicitly, altering their signatures.**
  → Implemented AsyncLocalStorage for PrismaTransactionClient. The TransactionMiddleware initiates prisma. and saves 'tx' via AsyncLocalStorage context. PrismaRepositories simply use getTx() || this.prisma, completely decoupling infra transactions from the domain interface.
- **Middlewares in CQRS CommandBus apply globally, causing unnecessary Transaction/Retry overhead on commands that don't need them.**
  → Added CommandOptions to ICommand interface (transactional, retryable). Each Command declares its own needs. Middlewares check command.options at runtime and call next() immediately if not opted in. Infra-specific logic (isPrismaTransientError, PrismaTransactionManager) is injected at composition root.
- **Không chủ động cập nhật tài liệu (SOP) ngay khi có quyết định kiến trúc mới (CQRS Pure POJO) mà chỉ tập trung code.** `[directives/memory_sop.md, CQRS Module]`
  → Đưa việc cập nhật file .md trong thư mục directives/ và log memory trở thành bước BẮT BUỘC và TỰ ĐỘNG ngay lập tức mỗi khi có rule mới, không đợi user nhắc.
- **Framework code (NestJS modules) nằm lẫn trong common/cqrs/ phá vỡ quy tắc Pure POJO của CQRS Bus.** `[common/cqrs, infrastructure/cqrs]`
  → Bắt buộc đẩy các class có @Injectable(), @Module() (ví dụ: cqrs.module.ts) ra khỏi common/ và đưa vào infrastructure/cqrs/. Các Bus trong common/ phải là Pure TypeScript.
- **Query handlers reusing Domain Repositories (UserRepository) violates strict CQRS by fetching Entities instead of DTOs, and creating unused methods like findById on Write models.** `[CQRS architecture in modules/<domain>]`
  → Strictly separate Query vs Command Repositories. Put Command repos in domain/repositories/ (returning Entities) and Query repos in application/repositories/ (returning DTOs). Flatten infrastructure/repositories/ without commands/queries subfolders.
- **Infrastructure Leakage: Query Handlers directly importing PrismaClient** `[GetProfileHandler using PrismaClient]`
  → CQRS Query Handlers must depend on QueryRepository interfaces defined in the application layer, not infrastructure details like Prisma.
- **Query logic duplicated across multiple Handlers and DTOs (e.g. get-profile vs get-me)** `[auth-service get-profile and get-me redundancy]`
  → Strictly enforce DRY for Query/Read Models. If two queries return the exact same DTO and use the same infra logic, merge them into a single Query/Handler.
- **Props interface pass-by-reference leak in Domain Entity constructors breaking OOP Encapsulation** `[Domain Entity Constructors]`
  → Always clone the props object in the Entity constructor (e.g. `this.props = { ...props }`) instead of using `private readonly props: EntityProps` directly as a parameter property, to prevent external reference leaks.
- **Anemic Domain Model: Handler created empty entity then called update() to fill data** `[DDD Entity Creation Lifecycle]`
  → Intention-Revealing Interfaces (Factory Pattern): Entity creation must be atomic. Use Factory methods on the Entity (e.g. `createPremiumUser`, `createForLogin`, or pass initial data directly to `create`) instead of creating a hollow entity and letting the Application layer (Handler) manipulate it. This prevents the Transaction Script anti-pattern.
- **Code Quality: Magic numbers used for token expiration times** `[auth-service token generation]`
  → Extract magic numbers into constants (e.g. ACCESS_TOKEN_TTL) in imp-token.service.ts
- **Code Quality: Accessing private entity properties via bracket notation (e.g., user['profile'])** `[auth-service user.mapper.ts and user.entity.ts]`
  → Implement explicit getter methods (getProfile(), getAuthMethods()) in the Entity class
- **Code Quality: Repository using fake setTimeout implementation instead of real DB calls** `[core-api Infrastructure repository implementation]`
  → Implement real Prisma calls in prisma-user.repository.ts and inject PrismaService
- **Used '-it' flag in docker exec causing TTY error. Skipped memory search before complex planning. Forgot to read directives/ SOPs before planning.**
  → Never use '-it' for automated docker exec scripts. Always run memory_manager.py search before making plans (Rule #2). Always read relevant SOPs in directives/ before creating implementation plans to properly cite them in 'References & Compliance' (Rule #3).
- **Exception Handling: Throwing generic Error in Handlers instead of Domain Errors** `[GetProfileHandler throwing new Error()]`
  → Always use custom errors inheriting from ApplicationError (e.g., UserNotFoundError) so global exception filters can catch them.
- **Schema and HTTP files poorly named (e.g., me.schema.ts, user.http) violating 1-1 mapping with Handlers** `[auth-service presentation layer naming]`
  → Schema files, HTTP test files, and Handlers must map 1-1 in naming convention (e.g., get-me.handler.ts -> get-me.schema.ts -> get-me.http). Do not group multiple requests into generic files.
- **Agent creating complex domain files before user explicitly approves the Implementation Plan (running ahead of the car)** `[Agent Etiquette during Expense Module planning]`
  → Always wait for explicit user approval (a clear 'yes' or 'approve') before executing any new Implementation Plan.
- **Domain entities declared using primitive properties in their constructor instead of a strongly typed Props interface, leading to parameter ordering bugs and lack of explicitness in rehydrate()/create().** `[Domain Entity structure across all modules (e.g. User, Permission, Role, Expense)]`
  → Always define a `[EntityName]Props` interface and use `private constructor(private readonly props: [EntityName]Props)` for Domain Entities. Use `Omit<Props, ...>` in factory methods like `create()` for clarity.
- **Inconsistent API routes regarding Swagger documentation, Authentication, and Zod Schema validation** `[Fastify Routing Conventions (fastify-type-provider-zod + fastify-swagger)]`
  → Follow Fastify Enterprise Pattern: Separate Swagger metadata (`description`, `tags`, `security: [{ cookieAuth: [] }]`) from Zod Validation (`...schema`). Place security enforcement in `preHandler` (e.g., `fastify.authenticate`). This ensures correct Swagger UI (locks) and proper Separation of Concerns.
- **Repeated Offense (3rd time): Throwing generic 'throw new Error()' in Application layer Handlers instead of specific Domain ApplicationErrors** `[RBAC Module Handlers]`
  → NEVER use generic 'throw new Error()' inside CQRS Handlers. You MUST create custom error classes (e.g., RoleAlreadyExistsError, PermissionNotFoundError) extending ApplicationError in src/common/errors/ and throw them. This guarantees global filters can catch and format HTTP responses correctly.
- **Inconsistent usage of deletedAt vs isActive causing ambiguity between 'Temporarily Disabled' and 'Soft Deleted'** `[Prisma Schema Design and Soft Deletion Strategy]`
  → Database Schema Convention: (1) Use 'isActive' for temporary toggle/disable (e.g. banning user, disabling role). (2) Use 'deletedAt' for Soft Delete (user-initiated delete, removing role permanently but keeping audit history). (3) Use Hard Delete (neither field) for mapping/join tables and transient data.
- **Entity creation used undefined instead of null, causing inconsistent mapping** `[Domain Entity Definitions]`
  → Entity Properties Nullability Convention: Entity properties mapping to DB columns MUST use `null` (not `undefined`) for optional/empty fields. This enforces explicit declaration of missing data. The `update()` method (Partial Update pattern) should use `?: type | null` where `undefined` means 'do not update' and `null` means 'clear the field'.
- **Prisma: Unique constraint failed on fields (userId, provider) instead of just userId** `[auth-service Prisma schema AuthMethod]`
  → Sửa @unique thành @@unique([userId, provider]) trong schema.prisma
- **Prisma: Missing DATABASE_URL env var during generation or runtime in auth-service and core-api** `[Prisma schema configuration]`
  → [SUPERSEDED by #16] Add `url = env("DATABASE_URL")` explicitly to the datasource block in schema.prisma
- **HTTP Protocol: Token validation failures returning HTTP 400 Bad Request** `[auth-service error handling]`
  → Change HTTP status code to 401 Unauthorized for all Token-related errors in auth.error.ts
- **Security: CORS configured with wildcard origin ['*']** `[auth-service fastify.ts security config]`
  → Load CORS origins from environment variables instead of hardcoding wildcard
- **Security/Runtime: fastify.jwt.decode() returning null causing crashes when accessing properties** `[auth-service JWT decoding in auth.routes.ts]`
  → Add null safety guard `if (!decoded?.sub)` before accessing token payload
- **DevOps: Services crash because Kafka or Elasticsearch haven't fully started** `[docker-compose dependency management]`
  → Add healthcheck configurations for Kafka and Elasticsearch in docker-compose.yml
- **Prisma schema validation - Error code: P1012: The datasource property url is no longer supported in schema files (Prisma v7)** `[Prisma v7 upgrade issue]`
  → Delete url=env() from schema.prisma and ensure prisma.config.ts has datasource: { url: process.env.DATABASE_URL }
- **Mở quyền Read-Write (:rw) cho Sandbox để tự động sửa lỗi, vô tình phá vỡ lớp bảo vệ an toàn Layer 0.** `[docker-compose.agent.yml, tool_builder_sop.md, Kiến trúc bảo mật]`

- **FST_ERR_LOG_INVALID_LOGGER_CONFIG: logger options only accepts a configuration object** `[apps/auth-service/src/bootstrap/server.ts]`
  → In Fastify v5, the 'logger' option only accepts a boolean or configuration object. To pass a custom Pino logger instance, use the 'loggerInstance' option instead of 'logger'.
- **FST_ERR_LOG_LOGGER_AND_LOGGER_INSTANCE_PROVIDED: You cannot provide both logger and loggerInstance** `[apps/auth-service/src/bootstrap/server.ts]`
